# docs.reearth.io 設計プロトタイプ

ドキュメントサイトを「読む場所」から「学ぶ場所」へ。docs.reearth.io の構成・導線・デザインを、学習設計の理論に基づいてプロトタイピングし、実物で検証した記録です。

**▶ デモ（GitHub Pages）: https://eukarya-biz.github.io/reearth-docs-prototype-/

---

## これは何か

Re:Earth プロダクトは、扱い始めるまでの認知的なハードルが高い。そのハードルを越えさせる学習インフラをつくることが Education の根幹ミッションであり、ドキュメントサイトはその中心に置かれる。

本リポジトリは docs.reearth.io の**設計を検証するためのプロトタイプ**であって、製品版ではない。実装は Astro + Starlight で別途行う。プロトは「満たすべき挙動と見た目」を示すもので、内部実装をそのまま移植する前提ではない。

## 何を検証したか（4つの設計理論）

「ドキュメントをプロダクトとして設計する」という前提に立ち、次の4理論がサイトの構成レイヤーで成立するかを検証した。

- **PAH 連続体** — ユーザーが自分の目的で学び始め、決められた順路をたどる段階から自分で目的地を選ぶ段階へ進めるよう、入口と導線を意図して配置する。
- **AI 読者対応** — 読者は人間だけではない。LLM・AI エージェントがセクション単位で正確に参照できる構造を、人間の読みやすさと両立させる。
- **インクルーシブ設計** — 複数のスキルレベル・背景の読者に複数の入口を用意し、キーボード操作・コントラスト等のアクセシビリティを満たす。
- **Docs as Product** — 書いて終わりにせず、更新日・古さの明示・フィードバック導線を備え、運用され改善され続ける状態にする。

## 主要な設計判断

| 判断 | 理由 |
|------|------|
| 製品ごとに入り、目的で選ばせる | 来訪動機の多くが「特定の製品を使いたい・評価したい」。各製品の入口を、やりたいこと（作る／理解する／操作する／調べる）で選べるハブにした。 |
| チュートリアルを独立させる | 手を動かして学ぶ導線を製品ドキュメントに埋め込まず、製品と対象読者で絞り込める一覧として独立させ、学習経路を俯瞰できるようにした。 |
| 種類は色でなく言葉で示す | コンテンツの種類（概念／手順／仕様／チュートリアル）を色だけに頼らずテキストで示す。実在の優れたドキュメントはすべてこの作法で、アクセシビリティ上も安全。 |
| 古さは「バージョン番号」でなく「鮮度」で伝える | Web アプリは機能がバージョン単位で管理されない。細かい版番号を各記事に出さず、「現行サービスから見て古い可能性」だけを伝える。 |
| コミュニティへの出口を持つ | 自律的な学習者はサイトの外（コミュニティ）で問い・共有し・つながることで学びを深める。その出口をトップに置く。 |

## どう検証したか

設計判断は思いつきではなく、実在するドキュメントサイトの調査で裏づけている。

- 実サイト調査6分野（PAH 移行／チュートリアル一覧／型の見分け／AI 対応／アクセシビリティ／運用）
- HashiCorp・Stripe・React・MDN・GOV.UK・Django・Kubernetes 等を実地確認
- プロトを評価基準で通し評価し反復

#### 突合マップ（○=拾えている / △=部分的 / ✗=欠落）

| 分類                 | 転用イメージ（§1.4.2原文の要旨）                 | 現状  | 対応                                                            |
| ------------------ | ----------------------------------- | --- | ------------------------------------------------------------- |
| 段階1 学習契約           | 目的別・スキル別の入口で「何を達成しに来たか」を選べる         | △   | 製品内 Overview を「目的に応じて選ぶ」ハブ化で対応。目的軸の入口は製品内どまり、スキル別入口はなし        |
| 段階2 学習活動           | 形式（テキスト/動画/インタラクティブ等）が役割を持ち、学習者が選べる | ✗   | E として保留（テキスト以外の整備方針が未策定）                                      |
| 段階3 学習成果アセスメント     | 「どこまでできたか」のセルフチェック（合否判定にしない）        | ✗   | B として設計待ち                                                     |
| 原則1 読み順の非強制        | 強制的な読み順をつくらず経路を選べる／変えられる            | ○   | タブ構造・自由遷移                                                     |
| 原則2 柔軟なカリキュラム      | ニーズが変われば必要コンテンツも変わる前提で入口・動線を配置      | △   | Overview ハブ・See also・前後ページ                                    |
| 原則3 参加型評価          | セルフチェック＋コミュニティでの成果共有・問答が評価の役割       | △   | コミュニティ側は A で対応。セルフチェック側は B 設計待ち                               |
| 原則4 個別化ナビ          | 「どの学習段階向けか」＋「これができたら次へ」             | △   | 「次のページ」「次にやってみよう」で"次へ"はある。"どの段階向けか"の本文明示は意図的に外した（下記メモ参照）      |
| 原則5 前段階との接続可視化     | いま見ているものと前の段階で学んだことの接続を可視化          | △   | 前後ページ・prereq ノート・See also                                     |
| 要素 探索(Explore)     | タグ・検索・関連リンク＋初学者向け「まずはここから」          | ○   | C で検索にファセット（製品×型）を追加。検索○/関連リンク○/「まずはここから」＝Overview ハブ・prereq○ |
| 要素 創造(Create)      | チュートリアル末尾の「作ってみよう」＋テンプレート・課題        | △   | 「次にやってみよう」○。テンプレ・課題は F として保留                                  |
| 要素 協働(Collaborate) | コミュニティ（SNS/GitHub/Discord等）への送り出し導線 | ○   | A で実装                                                         |
| 要素 つながり(Connect)   | 専門家・同分野の他者への送り出し導線                  | ○   | A で実装（コミュニティ導線が受け皿）                                           |
| 要素 共有(Share)       | 役立ったコンテンツの共有・ユースケース投稿の推奨と導線         | ✗   | D として保留（今は不要）                                                 |
| 要素 振り返り(Reflect)   | セルフチェック・振り返りの問い・学習の記録機能             | ✗   | セルフチェックは B 設計待ち、学習記録は G 保留                                    |
#### 対応方針
| ラベル | 項目                    | 判断       | 理由                                                   |
| --- | --------------------- | -------- | ---------------------------------------------------- |
| A   | コミュニティ導線              | **実装済**  | 協働・つながり・共有の一部が同じ「コミュニティへの送り出し」に収束する理論上の柱。今できることとして実装 |
| B   | セルフチェック               | **設計待ち** | 「ドキュメントサイトにおけるセルフチェックとは何か」を先に設計する。定義なしの実装はトークン・時間の無駄 |
| C   | 探索面の機能充実              | **実装済**  | 新規タグ体系を確定せず、既存メタデータ（製品／Diátaxis型）で検索ファセットを実現         |
| D   | 記事単位の共有ボタン・ユースケース投稿導線 | 保留       | 今は不要                                                 |
| E   | テキスト以外の形式             | 対象外      | 整備方針が未策定。今の設計に入れるものではない                              |
| F   | Create への接続（テンプレ・課題）  | 保留       | 何を Create させるとよいかの設計がない中で導線だけ考えても仕方がない               |
| G   | 学習記録・進捗機能             | 保留       | ログイン・バックエンド前提。静的ドキュメントサイト／CEOプロトの範囲外候補               |

## 構成

- `index.html` — 触れるプロトタイプ本体（GitHub Pages のトップ）。製品スイッチャー・チュートリアル一覧・検索・ダークモード・日英切替を操作できる。
- `README.md` — 本書。

デザイン・実装の詳細要求は要求仕様書に、設計理論はコンテンツ制作ガイドラインに定義している（別管理）。
