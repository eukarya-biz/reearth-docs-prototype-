# docs.reearth.io 設計プロトタイプ

ドキュメントサイトを「読む場所」から「学ぶ場所」へ。docs.reearth.io の構成・導線・デザインを、学習設計の理論に基づいてプロトタイピングし、実物で検証した記録です。

## これは何か

Re:Earth プロダクトは、扱い始めるまでの認知的なハードルが高い。そのハードルを越えさせる学習インフラをつくることが Education の根幹ミッションであり、ドキュメントサイトはその中心に置かれる。

本リポジトリは docs.reearth.io の**設計を検証するためのプロトタイプ**であって、製品版ではない。実装は Astro + Starlight で別途行う。プロトは「満たすべき挙動と見た目」を示すもので、内部実装をそのまま移植する前提ではない。

## 何を検証したか（4つの設計理論）

「ドキュメントをプロダクトとして設計する」という前提に立ち、次の4理論がサイトの構成レイヤーで成立するかを検証した。

- **PAH 連続体** — ユーザーが自分の目的で学び始め、決められた順路をたどる段階から自分で目的地を選ぶ段階へ進めるよう、入口と導線を意図して配置する。
- **AI 読者対応** — 読者は人間だけではない。LLM・AI エージェントがセクション単位で正確に参照できる構造を、人間の読みやすさと両立させる。
- **インクルーシブ設計** — 複数のスキルレベル・背景の読者に複数の入口を用意し、キーボード操作・コントラスト等のアクセシビリティを満たす。
- **Docs as Product** — 書いて終わりにせず、更新日・古さの明示・フィードバック導線を備え、運用され改善され続ける状態にする。

## 主要な設計判断

| 判断 | 理由 |
|------|------|
| 製品ごとに入り、目的で選ばせる | 来訪動機の多くが「特定の製品を使いたい・評価したい」。各製品の入口を、やりたいこと（作る／理解する／操作する／調べる）で選べるハブにした。 |
| チュートリアルを独立させる | 手を動かして学ぶ導線を製品ドキュメントに埋め込まず、製品と対象読者で絞り込める一覧として独立させ、学習経路を俯瞰できるようにした。 |
| 種類は色でなく言葉で示す | コンテンツの種類（概念／手順／仕様／チュートリアル）を色だけに頼らずテキストで示す。実在の優れたドキュメントはすべてこの作法で、アクセシビリティ上も安全。 |
| 古さは「バージョン番号」でなく「鮮度」で伝える | Web アプリは機能がバージョン単位で管理されない。細かい版番号を各記事に出さず、「現行サービスから見て古い可能性」だけを伝える。 |
| コミュニティへの出口を持つ | 自律的な学習者はサイトの外（コミュニティ）で問い・共有し・つながることで学びを深める。その出口をトップに置く。 |

## どう検証したか

設計判断は思いつきではなく、実在するドキュメントサイトの調査で裏づけている。

- 実サイト調査6分野（PAH 移行／チュートリアル一覧／型の見分け／AI 対応／アクセシビリティ／運用）
- HashiCorp・Stripe・React・MDN・GOV.UK・Django・Kubernetes 等を実地確認
- プロトを評価基準で通し評価し反復

- `README.md` — 本書。

デザイン・実装の詳細要求は要求仕様書に、設計理論はコンテンツ制作ガイドラインに定義している（別管理）。
